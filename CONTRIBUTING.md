# BE CONTRIBUTING

## 1. 계층 분리 규칙
- `api/`: 요청/응답 처리
- `schemas/`: DTO, validation 스키마
- `services/`: 유스케이스 중심 로직
- `repositories/`: DB 접근
- `clients/`: 외부 API/모델 서버 연동

## 2. 함수 작성 원칙
- 하나의 함수는 하나의 책임만 가집니다.
- 비즈니스 규칙은 라우터에 두지 않습니다.
- 예외는 가능한 한 명시적으로 처리합니다.

## 3. 추천 로직 규칙
추천 로직은 아래 순서를 기본으로 유지합니다.
1. 입력 검증
2. 재고/주류 컨텍스트 구성
3. 캐시 조회
4. 후보 조회
5. 점수화 및 랭킹
6. 부족 재료 계산
7. 설명 보완 생성
8. 결과 저장 및 반환

## 4. 스키마/응답 규칙
- 입출력 스키마는 명확히 분리합니다.
- FE가 바로 사용할 수 있도록 응답 구조를 일관되게 유지합니다.
- nullable 필드는 최소화합니다.

## 5. 테스트 권장 범위
- 이벤트 분기 테스트
- 재고 갱신 테스트
- 추천 후보 점수화 테스트
- 캐시 hit/miss 테스트
- 사용자 보정 반영 테스트

## 6. 금지 사항
- 라우터에 DB 쿼리 직접 작성하지 않기
- 환경 변수 값을 코드에 하드코딩하지 않기
- 예외를 무시하고 빈 값만 반환하지 않기
