# 추천 seed 선정 및 점수 정책

## 1. 목적
이 문서는 추천 레시피 seed를 어떤 기준으로 채택하고, 실제 추천 API가 어떤 기준으로 상위 3개를 고르는지 설명합니다.

핵심 목표는 “그럴듯한 AI 추천”이 아니라 “설명 가능한 추천 시스템”입니다.

## 2. 현재 추천 방식
현재 추천은 실시간 LLM 생성 방식이 아닙니다.

운영 흐름은 아래와 같습니다.

1. 검수된 seed 레시피를 DB에 적재합니다.
2. 감지된 주류를 내부 key로 정규화합니다.
3. 현재 냉장고 재고를 내부 식재료 key로 정규화합니다.
4. 해당 주류에 맞는 seed 후보만 조회합니다.
5. 각 후보의 핵심 재료가 현재 재고와 얼마나 맞는지 계산합니다.
6. 재고 매칭 점수와 기본 페어링 점수를 합쳐 상위 3개를 반환합니다.

즉 추천 API는 새 레시피를 즉석에서 창작하지 않고, 검수된 레시피 풀에서 현재 상황에 가장 잘 맞는 안주를 고릅니다.

## 3. seed 채택 기준
레시피가 `seeds/recommendations.json`에 들어가려면 아래 기준을 통과해야 합니다.

### 3-1. 지원 주류와 연결되어야 함
현재 seed는 아래 상위 주류 카테고리만 사용합니다.

- `soju`
- `beer`
- `red_wine`
- `white_wine`
- `sparkling_wine`
- `whisky`
- `sake`

서비스가 현재 상위 주류 카테고리만 인식하므로, 사용자에게 노출되는 문장에는 특정 하위 스타일을 단정하지 않습니다.

예를 들어 `카베르네 소비뇽`, `피노 누아`, `샴페인`, `프로세코`, `버번`, `싱글몰트`, `준마이` 같은 표현은 seed 생성 참고에는 쓸 수 있지만 최종 사용자 노출 문구에서는 제거합니다.

### 3-2. 허용된 핵심 식재료 key만 사용해야 함
핵심 재료는 내부에서 관리하는 식재료 key만 사용합니다.

예:

- `green_onion`
- `onion`
- `beef`
- `fish`
- `mushroom`
- `cheese`
- `pepper`

주의:

- `pepper`는 파프리카 의미로만 사용합니다.
- 청양고추, 홍고추, 풋고추 의미로 쓰지 않습니다.
- `fish`, `mushroom`, `cheese`는 `variant_detail`에 구체적인 종류를 적습니다.

### 3-3. 한국 1~2인 가구가 만들 수 있어야 함
레시피는 집에서 실제로 만들 수 있어야 합니다.

채택 가능한 예:

- 팬구이
- 볶음
- 찜
- 조림
- 샐러드/무침
- 수프/스튜
- 에어프라이어 구이
- 딥/오픈샌드

제외하는 예:

- 전문 장비가 필요한 요리
- 긴 숙성이나 복잡한 소스가 필요한 요리
- 와인바/이자카야 메뉴처럼 과장된 요리
- 냉장고 서비스의 현실성과 맞지 않는 요리

### 3-4. 조리 설명이 바로 따라 할 수 있어야 함
각 레시피는 최소한 아래 조건을 만족해야 합니다.

- `recipe_steps` 4단계 이상
- 각 step에 `title`, `instruction`, `time_minutes`, `heat_level`, `success_cue` 포함
- 초보자도 따라 할 수 있는 구체적인 instruction
- 조리 성공 여부를 알 수 있는 success cue
- 상온 양념과 핵심 재료의 구분

### 3-5. 서비스에 바로 노출 가능한 말투여야 함
사용자에게 보이는 문장은 부드러운 존댓말을 기본으로 합니다.

좋은 예:

- “냉장고 재료로 만들기 쉽고, 술 한 잔에 천천히 곁들이기 좋아요.”
- “팬을 충분히 예열하면 구운 향이 더 선명해져요.”

피하는 예:

- “~다.”로 끝나는 사전식 문장
- “주재료”, “나머지 재료”처럼 템플릿 티가 나는 문장
- “레드와인는”처럼 조사 오류가 있는 문장
- 실제 인식하지 못한 하위 주류 스타일을 단정하는 문장

### 3-6. 중복과 변형 남발을 피해야 함
이름만 바꾼 같은 조합은 seed에 넣지 않습니다.

예:

- “소고기 양파구이”
- “후추 소고기 양파구이”
- “버터 소고기 양파구이”

위처럼 핵심 구조가 거의 같으면 한두 개만 유지하고, 다른 조리 방식이나 다른 재료 조합을 우선합니다.

## 4. `rank_hint` 기준
`rank_hint`는 레시피 자체의 기본 페어링 점수입니다.

현재 범위는 60~100입니다.

- 90~100: 해당 주류와 매우 안정적인 대표 안주
- 80~89: 좋은 후보지만 재고 상황에 따라 우선순위가 바뀔 수 있는 안주
- 70~79: 다양성 확보용 보조 후보
- 60~69: 특수 상황이나 refresh 후보로만 쓸 수 있는 낮은 우선순위 후보

`rank_hint`는 최종 순위를 혼자 결정하지 않습니다.
현재 냉장고 재고와 얼마나 맞는지가 함께 반영됩니다.

## 5. 추천 점수 계산
추천 API는 현재 아래 공식으로 점수를 계산합니다.

```text
total_score = available_ingredient_count * 3
            - missing_ingredient_count * 2
            + rank_hint
```

의미:

- `available_ingredient_count`: 현재 재고에 있는 핵심 재료 수
- `missing_ingredient_count`: 현재 재고에 없는 핵심 재료 수
- `rank_hint`: seed에 저장된 기본 페어링 점수

예:

```text
핵심 재료 4개 중 4개 보유, rank_hint 94
total_score = 4 * 3 - 0 * 2 + 94 = 106
```

```text
핵심 재료 4개 중 2개 보유, 2개 부족, rank_hint 94
total_score = 2 * 3 - 2 * 2 + 94 = 96
```

따라서 추천은 “술과 잘 어울리는 안주”뿐 아니라 “지금 냉장고 재료로 만들 가능성이 높은 안주”를 우선합니다.

## 6. 추천 정렬 기준
후보는 아래 순서로 정렬합니다.

1. `total_score`가 높은 순서
2. 부족한 핵심 재료 수가 적은 순서
3. `rank_hint`가 높은 순서
4. 레시피 이름 오름차순

이 정렬 기준은 같은 점수 후보가 여러 개일 때 결과가 매번 흔들리지 않게 하기 위한 안전장치입니다.

## 7. `refresh` 동작
추천 API는 `refresh=false`일 때 현재 재고 기준으로 점수가 높은 상위 3개 후보를 먼저 반환합니다.

`refresh=true`일 때는 같은 정렬 결과에서 다음 3개 후보를 반환합니다.
후보 끝까지 도달하면 다시 앞쪽 후보로 순환합니다.

`refresh_group`은 seed 후보의 성격을 나누는 보조 메타데이터로 유지하지만, API 응답은 두 묶음에 고정하지 않습니다.
기존 `refresh_group=0`, `refresh_group=1` 값은 seed 검수와 후보 다양성 확인에 계속 사용합니다.

목표:

- 첫 추천은 안정적인 대표 후보 제공
- 다른 추천 보기는 가능한 한 새로운 후보 제공
- 후보 부족 상황에서도 빈 응답을 피함

## 8. `missing_ingredients` 원칙
`missing_ingredients`에는 사용자가 실제로 냉장고나 장보기에서 체감하는 핵심 재료만 넣습니다.

포함 예:

- 소고기
- 닭고기
- 양파
- 감자
- 버섯
- 치즈

제외 예:

- 소금
- 후추
- 식용유
- 간장
- 식초
- 설탕
- 마요네즈

상온 양념은 `pantry_items`로 따로 내려줍니다.

## 9. 사용자에게 설명하는 방식
추천 결과에서 설명은 아래 필드로 나뉩니다.

- `reason`: 이 안주가 해당 주류와 어울리는 이유
- `priority_reason`: 왜 지금 이 순위로 골랐는지
- `selection_factors`: 재고 매칭, 부족 재료, 페어링 점수 같은 선택 요인
- `score_breakdown`: 실제 점수 구성
- `missing_ingredients`: 부족한 핵심 재료

즉 FE는 추천 카드에서 “왜 이걸 추천했는지”와 “지금 만들 수 있는지”를 함께 보여줄 수 있습니다.

## 10. seed 확장 운영 방식
레시피를 늘릴 때는 아래 순서를 따릅니다.

1. 외부 LLM이나 사람이 후보 JSON을 생성합니다.
2. 내부 key, 허용 재료, pantry 구분을 검수합니다.
3. 조리 단계와 말투를 서비스 노출용으로 정리합니다.
4. 중복/변형 남발을 제거합니다.
5. 주류별 다양성과 `refresh_group` 균형을 확인합니다.
6. `seeds/recommendations.json`에 반영합니다.
7. 품질 게이트 테스트를 통과시킵니다.

현재 Codex의 역할은 후보를 그대로 믿는 것이 아니라, seed DB에 넣을 수 있는 형태로 검수하고 정제하는 것입니다.

## 11. 한 줄 설명
이 추천 시스템은 검수된 레시피 DB에서 감지된 주류와 현재 냉장고 재고에 가장 잘 맞는 안주를 점수화해 고르는 방식입니다.
