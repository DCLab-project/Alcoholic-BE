# 추천 지식 설계 문서

## 1. 문서 목적
이 문서는 `Alcoholic-BE`의 안주 추천 기능에 들어가는 지식 설계 원칙을 정리한 문서다.

나중에 다음 작업에 바로 재사용할 수 있도록 남긴다.

- 보고서/발표 자료 작성
- LLM 프롬프트 설계
- 레시피 seed 데이터 확장
- 추천 로직 고도화
- 사람 검수 기준 통일

## 2. 현재 추천 시스템 방향
현재 추천 시스템은 **DB 우선 + 점수화 + 필요 시 LLM 보완** 구조를 목표로 한다.

즉 운영 중 추천은 아래 순서로 동작한다.

1. 감지된 주류를 입력으로 받는다.
2. 현재 재고를 조회한다.
3. DB에 저장된 레시피 후보를 주류 기준으로 조회한다.
4. 현재 재고와 후보 레시피 재료를 비교한다.
5. 부족 재료 수와 사용 가능한 재료 수를 기준으로 점수화한다.
6. 상위 3개를 추천한다.
7. 나중에 필요하면 LLM으로 추천 이유를 더 자연스럽게 다듬거나 후보를 확장한다.

핵심은 다음과 같다.

- 추천 실행 시점에는 DB를 기준으로 움직인다.
- LLM은 운영 중 매번 전체 추천을 생성하는 주체가 아니다.
- LLM은 후보 데이터 구축과 설명 보완 역할로 쓴다.

## 3. 현재 DB seed 운영 기준
현재 seed는 `seeds/recommendations.json`에 저장한다.

현재 seed 규모:

- 주류 7종
- 각 주류당 50개 레시피
- 총 350개

주류 카테고리:

- `soju`
- `beer`
- `red_wine`
- `white_wine`
- `sparkling_wine`
- `whisky`
- `sake`

각 레시피는 현재 아래 필드를 가진다.

- `liquor_name`
- `refresh_group`
- `rank_hint`
- `name`
- `reason`
- `pairing_knowledge`
- `servings`
- `cook_time_minutes`
- `difficulty`
- `ingredient_yes`
- `ingredient_no`
- `ingredient_details`
- `pantry_items`
- `recipe_steps`
- `tip`
- `tags`

의미는 아래와 같다.

- `refresh_group`: 첫 추천과 다른 추천 보기 후보를 나누는 값
- `rank_hint`: 주류와 레시피의 기본 페어링 점수
- `pairing_knowledge`: 맛, 재료, 주류 관점의 페어링 설명
- `servings`: 기본 인분 수
- `cook_time_minutes`: 예상 조리 시간
- `difficulty`: FE에서 배지나 텍스트로 보여줄 난이도
- `ingredient_yes`: 현재 재고에 있어 바로 사용할 수 있는 필요 재료의 한글명 배열
- `ingredient_no`: 현재 재고에 없어 추가로 필요한 필요 재료의 한글명 배열
- `ingredient_details`: 내부 key + 한글명 + 정량 + 단위
- `pantry_items`: 상온 기본 보유 양념/재료
- `recipe_steps`: 상세 조리 단계
- `tip`: 사용자가 바로 이해할 수 있는 짧은 조리 팁
- `tags`: FE 필터나 배지에 활용할 태그

즉 현재 추천 DB는 단순 카드용 텍스트를 넘어서, **상세 레시피 화면까지 확장 가능한 구조**를 목표로 한다.

seed 선정 기준과 추천 점수 정책은 `docs/recommendation_policy.md`에 별도로 정리한다.

추가로 2026-04-24 기준 추천 응답에는 아래 우선순위 설명 필드도 포함한다.

- `priority_rank`
- `priority_reason`
- `selection_factors`
- `score_breakdown`

이 필드들은 FE에서 “왜 이 3개가 이 순서로 추천됐는지”를 보여줄 때 사용한다.

## 4. pantry(상온 양념) 가정
추천 시스템은 **상온 양념 재료는 기본 보유**한다고 가정한다.

예시:

- 소금
- 후추
- 간장
- 식용유
- 식초
- 설탕
- 고춧가루
- 기본 소스류

따라서 `missing_ingredients`에는 되도록 아래처럼 **냉장고/보관 재료로 체감되는 핵심 재료** 위주로 들어가야 한다.

- 육류/생선
- 달걀
- 채소
- 유제품
- 소시지/베이컨
- 빵

즉 seed의 `ingredients`는 "없으면 실제로 냉장고에서 꺼내야 하는 재료" 중심으로 유지한다.

로컬 데모 테스트에서는 `scripts/seed_demo_inventory.py`로 냉장고를 넉넉하게 채운 상태를 먼저 만들고 추천을 확인한다.

## 5. 추천 품질 검수 기준
레시피를 `approved` 수준으로 보기 위한 핵심 기준은 아래와 같다.

1. 해당 주류와 실제로 어울리는가
2. 한국 사용자 기준으로 납득되는가
3. 집에서 만들 수 있는 난이도인가
4. 현재 프로젝트 식재료 클래스와 연결 가능한가
5. 추천 이유가 사용자가 이해할 수 있을 만큼 설득력 있는가
6. 추천 3개가 서로 지나치게 비슷하지 않은가

검수 판단 기준:

- `유지`: 그대로 써도 무난함
- `수정`: 방향은 맞지만 더 좋은 대체안이 있음
- `제외`: 페어링 설득력이 약하거나 서비스 톤과 맞지 않음

## 6. 주류별 페어링 원칙

### 6-1. soju
기본 방향:

- 짭짤함
- 볶음
- 전
- 무침
- 돼지고기/소시지/달걀/대파 같은 익숙한 한식형 안주

설명 원칙:

- 소주의 직선적인 맛을 눌러주거나
- 느끼함을 잡아주는 조합을 강조한다.

### 6-2. beer
기본 방향:

- 감자
- 치즈
- 베이컨
- 소시지
- 닭고기
- 토스트/구이류

설명 원칙:

- 탄산감과 짭짤함, 고소함, 바삭함의 궁합을 강조한다.

### 6-3. white_wine
기본 방향:

- 생선
- 닭고기
- 치즈
- 요거트
- 가벼운 샐러드
- 버터를 곁들인 담백한 조리

설명 원칙:

- 산미
- 담백함
- 부드러운 질감

이 세 가지의 균형을 강조한다.

### 6-4. red_wine
기본 방향:

- 소고기
- 버섯
- 토마토
- 치즈
- 가지
- 베이컨

설명 원칙:

- 육향
- 버섯의 우마미
- 토마토 산미
- 진한 풍미

처럼 더 묵직한 조합을 강조한다.

주의:

- 너무 가벼운 샐러드형 메뉴는 우선순위를 낮춘다.

### 6-5. whisky
기본 방향:

- 베이컨
- 소고기
- 소시지
- 치즈
- 버터
- 버섯

설명 원칙:

- 훈연감
- 짭짤함
- 묵직함
- 진한 풍미

와 연결한다.

주의:

- 너무 산뜻하거나 채소 중심인 메뉴는 우선순위를 낮춘다.

### 6-6. sparkling_wine
기본 방향:

- 한입 크기 메뉴
- 샐러드
- 치즈
- 오이/토마토
- 가벼운 토스트
- 산뜻한 조합

설명 원칙:

- 청량감
- 산뜻함
- 가벼운 치즈/채소 조합

을 강조한다.

주의:

- 너무 묵직하고 기름진 메뉴는 우선순위를 낮춘다.

### 6-7. sake
기본 방향:

- 생선
- 달걀
- 버섯
- 대파
- 닭고기
- 담백하고 우마미 있는 조합

설명 원칙:

- 섬세한 향
- 우마미
- 자극적이지 않은 조리

를 강조한다.

주의:

- 토마토 산미가 강하거나 너무 서양식으로 치우친 조합은 우선순위를 낮춘다.

## 7. 2026-04-24 1차 큐레이션 변경 내용
이번 1차 검수에서 특히 수정 우선순위가 높았던 카테고리는 아래 4개였다.

- `red_wine`
- `whisky`
- `sparkling_wine`
- `sake`

교체한 항목:

- `red_wine`
  - `치즈 브로콜리그라탱풍 볶음` -> `버섯 치즈구이`
  - `토마토 양파샐러드` -> `베이컨 토마토볶음`
- `whisky`
  - `베이컨 브로콜리볶음` -> `베이컨 양파볶음`
  - `브레드 치즈플레이트풍 토스트` -> `소시지 치즈구이`
- `sparkling_wine`
  - `감자 치즈바이트` -> `치즈 오이 카나페풍 토스트`
  - `요거트 토마토브레드` -> `토마토 치즈 카나페풍 토스트`
- `sake`
  - `양배추 닭볶음` -> `닭고기 버섯볶음`
  - `생선 토마토조림풍 볶음` -> `생선 버섯구이풍 볶음`
  - `당근 양배추볶음` -> `양배추 버섯볶음`

수정 이유는 공통적으로 아래에 가깝다.

- 주류와의 거리감이 더 가까운 메뉴로 교체
- 너무 가벼운/어색한 메뉴 제거
- 서비스 설명력이 더 높은 메뉴로 교체

## 8. 앞으로의 운영 방식
현재 seed는 `approved` 후보 집합처럼 다루되, 장기적으로는 아래 상태 관리가 있으면 좋다.

- `draft`: LLM 초안
- `reviewed`: 사람이 1차 검토
- `approved`: 서비스 노출 가능
- `archived`: 제외

향후 1000개 규모까지 확장할 때는 반드시 위 상태 관리가 필요하다.

## 9. 상세 레시피 설계 원칙
상세 레시피는 “전문 셰프 레시피”가 아니라 **냉장고 앞 사용자가 바로 따라할 수 있는 1인분 중심 요약 레시피**를 목표로 한다.

원칙은 아래와 같다.

1. 재료는 되도록 정량으로 표현한다.
2. 단계는 3~4단계 안에서 끝나도록 유지한다.
3. 조리 시간은 10~20분 안쪽의 현실적인 값으로 잡는다.
4. 상온 양념은 `pantry_items`로 따로 빼고, `missing_ingredients`에는 냉장고 핵심 재료만 남긴다.
5. 레시피 문장은 초보자도 바로 따라할 수 있을 정도로 구체적으로 쓴다.

좋은 문장 예시는 아래와 같다.

- `대파 1대는 4~5cm 길이로 썬다`
- `팬에 기름을 두르고 돼지고기 180g을 먼저 2~3분 볶는다`

반대로 아래처럼 너무 추상적인 문장은 피한다.

- `재료를 손질한다`
- `적당히 볶는다`

## 10. 현재 우선순위 산정 규칙
현재 추천 우선순위는 아래 값으로 계산한다.

- `available_count * 3`
- `missing_count * -2`
- `rank_hint`

즉 수식으로 쓰면 아래와 같다.

`total_score = (보유 재료 수 x 3) - (부족 재료 수 x 2) + rank_hint`

정렬 규칙은 다음 순서다.

1. `total_score`가 높은 순
2. `missing_ingredient_count`가 적은 순
3. `rank_hint`가 높은 순
4. 이름 오름차순

이 규칙은 아직 1차 버전이며, 나중에는 아래 요소를 추가로 붙일 수 있다.

- 최근 본 추천 중복 회피
- 현재 재고 소진 우선도
- 조리 시간 가산/감점
- FE에서 선택한 사용자 선호도
- LLM 기반 설명 보완
